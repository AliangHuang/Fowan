# Fowan 公开仓库边界

## 1. 仓库职责

本仓库包含 GPL-3.0 许可的原生客户端、普通工具功能、平台集成、设计资源、构建脚本和公开产品文档。

以下内容属于本仓库：

- Windows 原生窗口、托盘、快捷键、通知、启动和安装体验。
- Todo、Diary 等工具的界面与普通业务功能。
- 工具共享模型、本地 JSON 数据格式、查询、回收站、附件、定位和天气集成。
- 客户端设置、可访问性、视觉规范、测试夹具和开发脚本。

目录或程序集名称中的 `Shared` 表示同一工具多个客户端窗口共享的开源代码，不表示 Fowan 的闭源核心。

## 2. FowanCore 职责

以下能力只能在私有 `FowanCore` 仓库实现：

- AI 工作流编排、模型选择、提示策略和商业化 AI 调度。
- 用户信息加密、解密、密钥生成、密钥保存策略与轮换。
- 敏感内容索引、敏感数据分类和访问控制策略。
- 授权判定、商业能力开关、同步冲突策略及其他需要保护的核心实现。

公开客户端只能展示这些能力的状态、收集用户选择，并通过版本化公开协议调用它们。不得把私有实现复制到 UI、`Shared` 项目、脚本或公开测试中。

## 3. 当前阶段

当前 Todo、Diary 和 Toolbox 不依赖 FowanCore，公开仓库可以独立构建并完整运行现有功能。

AI 工具是首个真实的私有核心接入能力。其公开、版本化契约位于 `protocol/ai/v0.1`；Windows 客户端只负责配置输入、明确的云端发送确认、流式结果展示和取消。公开仓库中的 `Fowan.Ai.Shared` 只包含 DTO、JSON-RPC 客户端、安全错误摘要、Core 定位和跨应用启动逻辑，AI 对话与 AI 配置中心的 UI 分别位于独立 WinUI 项目。API 密钥、模型策略、渠道调用、对话编排及加密历史由独立的 FowanCore artifact 实现。公开仓库不引用或编译私有源码，Release 打包必须显式提供 Core artifact。

AI v0.1 已是首个获批的私有能力接入：公开仓库先定义最小、版本化且与实现无关的协议，FowanCore 再实现该协议。后续能力仍不得创建无真实用例的占位 RPC、复制私有实现，或绕过兼容性与许可证评估。

## 4. 变更检查

新增功能前必须回答：

1. 它是否只是原生 UI、平台适配或普通工具行为？如果是，留在 Fowan。
2. 它是否包含 AI 策略、用户信息加解密、密钥或敏感商业规则？如果是，进入 FowanCore。
3. 客户端是否只需要稳定数据契约，而不需要知道实现细节？如果是，在真实用例出现时新增公开协议。
4. 是否改变现有本地数据格式或位置？如果是，必须提供迁移和兼容测试。

## 5. 分发与许可

Fowan 当前使用 GPL-3.0。未来若与闭源 sidecar 一同分发，必须在发布前专项评估两者的通信语义、组合方式、许可证兼容性和用户权利；使用独立进程或 IPC 本身不构成自动豁免。
