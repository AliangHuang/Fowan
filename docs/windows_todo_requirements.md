# Fowan Windows 待办需求基准

> 范围：独立 Windows 待办工具、主窗口、精致桌面便签模式。
> 视觉参考：`docs/windows_todo_ui_design.md`

## 1. 产品定位

Fowan Windows 待办是独立的 Windows 待办工具，不是工具箱主界面的卡片详情页。工具箱可以把待办作为入口打开，但待办工具必须保持独立进程、独立窗口、独立数据文件和独立构建入口。

待办工具只承载任务管理能力，不扩展到邮箱、购物、专注、统计、协作或日历套件。

## 2. 功能范围

本地可用版必须支持：

- 新增、编辑、删除任务。
- 完成任务和恢复任务。
- 任务必须维护树状关系，每个任务可以创建子任务。
- 带有子任务的父任务必须支持展开/收起子任务，折叠状态需要持久化，并在主窗口和便签模式中保持一致。
- 完成带有未完成子任务的父任务时，必须先弹窗提示“是否完成当前任务，仍有未完成的子任务”；用户确认后完成当前任务和所有子任务，用户取消时不能修改任何任务状态。
- 便签模式下必须支持长按单个任务后拖动，改变任务显示位置或父子任务关系；拖动落点仍必须遵循任务树最多三层、每个任务最多 100 个直接子任务的全局限制。
- 便签模式拖动任务时，必须在目标位置生成预览虚框；虚框位置和缩进必须表达松手后的具体归属，包括同级前后位置或作为目标任务子任务，避免用户需要反复试错。
- 便签模式拖动任务时，如果鼠标位置位于两个任务之间的空白区域，必须视为与当前下方任务平级，并在下方任务之前显示预览虚框。
- 便签模式拖动任务时，如果鼠标位于父任务下边缘且下方紧邻可见子任务，必须延续两个任务之间的规则，按下方子任务的缩进和父子归属显示预览虚框。
- 便签模式拖动任务时，如果鼠标位于当前分组最底部任务和底线之间，必须视为放到顶层任务末尾，预览虚框使用顶层缩进。
- 便签模式拖动任务成为另一个任务的子任务时，必须默认排在该父任务的第一个子任务位置，避免子任务较多时用户看不到创建效果。
- 便签模式拖动任务时，只有理论落点的纵向位置和缩进/父子归属都与原位置一致时，才不能显示预览虚框，也不能执行无意义的位置变更。
- 便签模式拖动任务时，如果理论落点的纵向位置不变，但缩进或父子归属会变化，仍然必须显示预览虚框，且虚框缩进必须表达松手后的实际归属。
- 任务树最多三层；每个任务最多 100 个直接子任务。
- 标记重要任务。
- 开始时间，默认值为当天。
- 可选截止日期；在主输入栏输入文字后按回车新增任务时，默认没有截止日期；主界面新建任务弹窗中的截止日期也必须默认为空。
- 清单分类可新增、删除、改名，并可从蓝宝石、孔雀青、石板蓝、紫晶、玫瑰、珊瑚、琥珀、祖母绿、深海蓝、雾紫、橄榄、铜棕十二种低饱和宝石色中选择；默认清单也可改色。
- 默认清单。
- 今日任务、计划任务、重要任务、全部任务、已完成、清单视图。
- 已完成任务弱化显示，并带删除线。
- 今日任务视图中的已完成分组只显示当天完成的任务，历史已完成任务不再显示在今日任务中；历史完成记录仍通过独立已完成视图查看。
- 本地 JSON 持久化，且数据文件必须保存到默认用户数据根目录 `%LOCALAPPDATA%\Fowan` 下的待办对应目录，不能保存到程序输出目录、安装目录或临时目录。

明确不做：

- 收件箱。
- 购物清单专属入口。
- 归档。
- 专注时长或番茄钟。
- 时间统计、效率分析或看板。
- 协作。
- 邮箱功能。
- 日历套件。

## 3. 主窗口需求

软需求/默认选型：主待办窗口优先继续使用 WinUI 3 / Windows App SDK。该选型服务于现有主窗口实现和概念图对齐，但优先级低于数据正确性、独立运行、主题一致性和便签硬需求。

主窗口必须支持：

- 浅色主题、深色主题、跟随系统。
- 布局、层级和视觉气质对齐现有待办概念图。
- 左侧导航只包含待办视图和用户清单。
- 中间区域包含任务输入、未完成任务列表和已完成分组。
- 中间任务列表必须支持和便签模式一致的长按拖动排序、父子关系调整、虚线落点预览和非法落点保护。
- 右侧详情面板包含任务标题、完成状态、重要标记、所属清单、开始时间、截止日期、备注、恢复和删除操作；已完成任务还必须支持编辑完成日期和完成时间。
- 主窗口任务右键菜单与便签保持一致，均提供“创建子任务”和“删除任务”；回收站任务不显示该菜单。
- 主窗口顶部全宽的无交互区域和侧栏品牌区域必须可拖动窗口；筛选、便签切换、侧栏导航/清单菜单、输入框、任务卡及其操作按钮、系统窗口控件必须保持可交互，不能被标题栏拖动命中覆盖。
- 设置入口包含主题切换和便签模式入口。
- 主界面左下角问号必须作为操作说明入口；点击后弹出主题一致、面向最终用户的说明窗口，分别介绍主界面模式和便签模式当前支持哪些功能、如何操作和关键使用限制，不展示数据结构或底层技术设计。

## 4. 便签模式需求

便签模式是主待办窗口的紧凑桌面形态，不是新的任务模块。

便签模式必须支持：

- 独立便签窗口启动。
- 置顶切换。
- 回到大界面。
- 关闭便签。
- 新增任务。
- 完成任务和恢复任务。
- 已完成分组展开和折叠。
- 持久化当前视图、选中任务、置顶状态、透明度、UI 缩放、窗口位置和窗口大小。
- 如果上次退出时处于便签模式，再次启动待办时直接进入便签模式。
- 启动无闪烁是硬需求：如果启动配置保存的是便签模式，应用启动时必须直接打开便签 shell，不能先显示主窗口再切换到便签模式。
- 主窗口和便签模式切换必须复用隐藏实例或已运行实例，首次切换不能出现明显冷启动卡顿；切换后不能留下不可见的孤儿待办进程。

便签窗口体验要求：

- 窗口真透明是硬需求：调整透明度时必须能真实透出桌面/底层窗口，不能只是降低窗口亮度、给背景加 tint、或让整窗内容发灰。
- 边缘缩放是硬需求：四边和四角必须支持稳定拖拽缩放。
- 硬需求：便签窗口边缘拖拽命中范围必须限制在 8-12 像素之间，既不能低于 8，也不能高于 12。
- 圆角不是硬需求：如果真透明和边缘缩放与圆角冲突，可以牺牲圆角。
- 系统阴影不是硬需求：如果真透明方案无法保留系统阴影，可以使用轻量自绘阴影或取消阴影。
- 四边和四角显示系统缩放光标。
- 四边和四角可拖拽缩放。
- 标题/品牌区域可拖动窗口。
- 支持动态透明度调整，且不能完全透明导致窗口不可见。
- 支持 50%-200% UI 等比整体缩放。
- 硬需求：便签透明度和 UI 缩放必须分别提供一键重置按钮，两个重置按钮都将对应值恢复到 100%。
- 硬需求：便签模式缩放后的窗口宽高不能超过当前所在显示器的分辨率。
- 硬需求：便签窗口最小尺寸限制必须跟随当前 UI 缩放设置同比例调整。
- 硬需求：便签窗口的品牌/图标拖动区域必须始终处于可见显示器范围内；多显示器场景下也必须处于任一显示器的显示范围内，不能因为该区域出界导致鼠标无法拖回窗口。
- 硬需求：便签窗口位置纠正不能发生在窗口拖动过程中；拖动品牌/图标区域到显示器边缘外时，不允许立刻夹回导致抖动，必须等鼠标松开或系统 move/resize 循环结束后再纠正位置。
- 软需求：透明度和缩放控制界面优先使用独立弹窗/浮层形式呈现，尽量不常驻占用便签主界面空间。
- 硬需求：便签模式下的弹出窗口必须被视为便签整体的一部分，和主便签窗口同步透明度、UI 缩放、置顶、主题和位置关系；后续新增的便签子窗口和配置窗口也必须遵守同一约束。
- 硬需求：便签模式同一时间最多只能弹出一个子窗口；打开新的子窗口前必须关闭当前子窗口。
- 硬需求：便签模式下存在子窗口时，便签主窗口内容层必须显示一层透明蒙板；点击蒙板必须关闭当前子窗口。
- 硬需求：便签子窗口不能被独立拖动；移动便签主窗口时，当前子窗口必须一起移动并保持相对位置。
- 硬需求：便签新增任务小窗显示在便签窗口正中；透明度、缩放和主题控制窗口显示在便签窗口中上方。
- 硬需求：便签子窗口必须在显示前完成尺寸、透明度、缩放和目标位置计算，第一次显示时就位于需求位置；不允许先创建显示在默认位置，再移动到目标位置。
- 硬需求：便签透明度、缩放和主题控制窗口必须支持切换主题。
- 硬需求：点击便签添加任务输入框后必须显示输入光标。
- 硬需求：便签添加任务输入框为空时点击加号，必须弹出风格一致的新增任务小窗。
- UI 缩放是硬需求：便签模式下字体、图标、间距和控件可以跟随窗口整体缩放，但必须保持等比缩放，不能出现横向或纵向单独拉伸导致的变形。
- 主题一致性是硬需求：所有窗口的 UI 控件，包括主窗口、便签窗口、弹窗、菜单、滚动条、滑块、输入框、按钮、下拉框和托盘菜单，都必须和当前浅色/深色/跟随系统主题保持一致。
- 无双光标闪烁。
- 无顶部黑边或白边。

## 5. 技术边界

主待办窗口继续使用 WinUI 3 / Windows App SDK 是软需求和默认技术选型，不是高于便签硬需求的限制。

精致便签版不再使用纯 WinUI `StickyTodoWindow` 承担窗口外壳。便签窗口同时需要隐藏系统按钮、DWM 圆角、透明度、UI 缩放、边缘系统缩放光标、边缘拖拽缩放和无可见边框瑕疵；在 WinUI `Window`、WndProc hook、XAML 命中区域和 DWM 属性之间反复组合，已经造成光标冲突和缩放行为不稳定。

精致便签版技术决策更新：

- 真透明和边缘缩放优先级高于圆角、DWM 阴影和 WindowChrome 外观一致性。
- 如果 WPF `WindowChrome` 无法同时满足真透明和边缘缩放，则不能继续把它作为最终便签外壳。
- 可接受的实现方向包括 WPF layered window + 自定义命中测试、原生 Win32 layered window、自定义 `WM_NCHITTEST` 边缘缩放，或其他能同时满足真透明和边缘缩放的独立 shell。
- 不接受只用 `Window.Opacity` 降低整窗透明度的方案，因为它会让文字和控件一起变淡，表现为亮度变化而不是真透明便签。
- WinUI 主窗口和 WPF 便签窗口共同引用待办 core。
- 待办 core 负责模型、设置、数据文件路径和 JSON 读写。
- 待办 core 的数据路径必须从默认用户数据根目录 `%LOCALAPPDATA%\Fowan` 派生，并落在待办对应子目录；主窗口、便签窗口和后续工具箱入口必须读写同一位置。
- 当前迭代中，主窗口和便签窗口通过共享 JSON 数据和设置同步，不新增复杂实时 IPC。

## 6. 数据与设置

数据存储硬需求：

- 待办数据必须保存到默认用户数据根目录 `%LOCALAPPDATA%\Fowan` 下的待办对应目录。
- 待办数据路径必须由共享 core 统一提供，主窗口和便签窗口不能各自拼接不同路径。
- 数据不能写入 `AppContext.BaseDirectory`、构建输出目录、安装目录或其他会随升级/清理丢失的位置。

数据文件：

- `%LOCALAPPDATA%\Fowan\Todo\todo-data.json`
- `%LOCALAPPDATA%\Fowan\Todo\todo-settings.json`

最小数据模型：

- `TodoTask`: `Id`, `Title`, `Notes`, `ListId`, `ParentTaskId`, `SortOrder`, `IsImportant`, `StartDate`, `DueDate`, `IsCompleted`, `CompletedAt`, `DeletedAt`, `CreatedAt`, `UpdatedAt`
- `TodoList`: `Id`, `Name`, `ColorId`, `CreatedAt`
- `TodoData`: `SchemaVersion`, `Lists`, `Tasks`
- `TodoSettings`: 主题、当前视图、选中任务、已完成分组状态、便签模式状态、置顶、透明度、UI 缩放、窗口位置、窗口大小、回收站开关、回收站清理预设、自定义清理天数

## 7. 回收站、筛选与便签补充

- 回收站默认开启，默认自动清理周期为 30 天；设置中可关闭回收站，或选择 7 天、30 天、90 天和 1-365 天自定义周期。
- 删除任务必须按整棵任务树处理。启用回收站时，父任务、子任务和孙任务一起软删除并保留层级；关闭回收站时，后续删除直接永久删除，但不主动清空既有回收站内容。
- 回收站入口位于主窗口左侧“已完成”下方，并支持整棵任务树恢复和永久删除。启动、刷新、保存和每小时维护时清理超过保留期的回收站任务树。
- 主窗口和便签的删除操作共用同一树删除逻辑。删除含有后代的任务时必须明确提醒用户，说明会同时删除所有子任务和孙子任务，并等待手动确认。
- 主窗口筛选支持任务清单、任务层级范围和日期条件的组合：清单支持全部清单和任一现有清单；层级支持只显示一级、一至两级、全部显示；日期筛选支持开始日期和执行周期模式，日期区间为闭区间，并提供本周、本月快捷项。日期模式为“不按日期筛选”时隐藏日期输入；快捷项未预先选择日期模式时默认启用执行周期，已明确选择开始日期时保持该选择。清单或日期条件应用后切换到全部任务，切换左侧导航时清除清单和日期条件；筛选生效时按钮显示“筛选中”，清空全部筛选后恢复“筛选”。
- 应用日期筛选时必须切换到“全部任务”。切换任意导航视图时清除日期筛选，因此“今日任务”页面不展示日期筛选结果。
- 今日任务中的已完成分组只显示当天完成的任务。便签模式无论从哪个主窗口视图进入，都固定按今日任务逻辑显示，不覆盖主窗口保存的原视图。
- 主窗口和便签拖拽任务时，指针靠近列表顶部或底部边缘必须自动上下滚动；仅按纵向距离判断，速度线性变化并设置上限，不根据水平距离触发滚动。自动滚动必须按未完成和已完成分组分别限界，但在边界保留 50 DIP 的可见重叠区；普通手动滚动不受此限制，拖拽落点仍不得跨完成状态。
- 主窗口创建子任务弹窗必须支持标题和多行备注；新子任务插入父任务的第一个子项位置并自动展开父任务。
- 便签模式右键任务菜单至少包含“删除任务”和“创建子任务”；菜单不得出现系统默认白色表头，背景、边框、悬浮态和透明度必须适配便签当前主题。创建子任务使用便签子窗口体系，支持标题和多行备注，并遵守三层与每父任务 100 个直接子项限制。
- 便签中左键双击任意可见任务时，打开可编辑标题和多行备注的任务详情子窗口；只有保存会写入，取消、关闭和 Esc 都丢弃草稿。该窗口遵守便签子窗口的主题、透明度、缩放、置顶和位置同步规则。
- 主窗口和便签任务卡鼠标悬浮时必须显示与任务卡一致的低对比圆角待选高亮，浅色和深色主题下文字均保持可读，选中和拖拽结束后不能丢失或退化为直角白色外框。
- 已完成任务编辑完成日期或时间时，必须显示应用修改和撤销修改；两个编辑器均为 236 DIP 宽，详情布局必须为其保留完整可操作区域；仅应用修改会写入完成时间，切换任务、视图或关闭详情时自动丢弃未应用草稿。
- 帮助弹层在窗口最小尺寸、缩放和主题变化后必须保持右上关闭按钮内的 X 图标完全居中、右下确认按钮四角完整可见；正文区域单独滚动。

## 8. 验收标准

- 主待办窗口可以独立于工具箱 UI 构建和运行。
- 主待办窗口和便签窗口读写同一份待办数据。
- 待办数据和设置文件位于默认用户数据根目录 `%LOCALAPPDATA%\Fowan` 下的待办对应目录。
- 在便签模式新增、完成或恢复任务后，回到主窗口能看到一致数据。
- 进入便签模式时打开精致便签 shell。
- 如果启动配置为便签模式，启动过程不显示主窗口，不出现主窗口闪烁后再切换便签的现象。
- 主窗口和便签模式之间连续切换时应保持同一主窗口进程和同一便签进程复用；退出任一可见模式时必须清理对应隐藏协调进程，不能留下后台孤儿。
- 回到大界面时恢复上次视图。
- 主界面左下角问号可以打开操作说明；说明内容必须覆盖主界面模式和便签模式，只保留用户需要的功能介绍和操作方法，不引入待办工具范围外的功能说明，也不展示数据结构或底层设计。
- 透明度调整能真实透出桌面/底层窗口，文字和控件保持可读。
- 透明度和缩放控制优先以独立弹窗/浮层呈现，便签主界面不应长期显示设置面板。
- 便签弹窗与便签主窗口同步透明度、缩放、置顶、主题和位置关系；新增任务小窗和后续配置子窗口不能脱离便签整体。
- 便签模式最多只允许一个子窗口；新增任务小窗居中，控制窗口位于中上方，子窗口随便签主窗口移动且不可独立移动。
- 便签子窗口打开时，便签主窗口显示透明蒙板；点击蒙板关闭当前子窗口，点击弹窗内部不关闭。
- 便签子窗口在调用 `Show()` 前完成目标位置计算，不出现默认位置闪现后再挪动的过程。
- 便签控制窗口提供主题切换，主题切换后主便签和当前子窗口同步更新。
- 便签控制窗口提供透明度重置和缩放重置；透明度重置到 100%，缩放重置到 100%。
- 点击便签添加任务输入框显示输入光标；空输入点击加号时显示新增任务小窗。
- 便签模式下使用缩进展示子任务层级，最多展示三层树状任务。
- 带有子任务的父任务可展开/收起，折叠状态重启后仍保持；便签模式下同样按折叠状态显示子任务。
- 完成带有未完成子任务的父任务时，弹出确认；确认会完成父任务和所有子任务，取消不会改变任何任务状态。
- 主窗口和便签模式下长按任务后均可拖动排序；拖到其他任务边缘可调整同级前后位置，拖到其他任务主体区域可设为其子任务；非法落点不能修改数据。
- 主窗口和便签拖动过程中必须显示虚线预览框；预览框的插入位置、缩进和提示文本能够说明松手后任务会归属到哪个父级或顶层位置。
- 拖动到两个任务之间时，虚线预览框显示在下方任务之前，缩进与下方任务一致，松手后任务成为下方任务的同级任务。
- 拖动到父任务下边缘且下方紧邻可见子任务时，虚线预览框也必须沿用下方子任务的缩进，不能跳回父任务同级缩进。
- 拖动到分组最底部任务与底线之间时，虚线预览框显示在底部空隙，缩进为顶层任务，松手后任务成为顶层末尾任务。
- 拖动到任务主体区域并设为其子任务时，虚线预览框显示在父任务下方、现有子任务之前，松手后任务成为该父任务的第一个子任务。
- 拖动只改变同一父级内的前后顺序时，任务和其子任务必须保持原清单不变；只有父子关系发生变化时，才允许按新的父任务归属同步整棵子树的清单。
- 只有拖动落点的纵向位置和缩进/父子关系都等同于任务当前状态时，虚线预览框才不显示，松手后任务位置和父子关系保持不变。
- 如果拖动落点的纵向位置看起来没有变化，但缩进或父子关系会变化，必须显示虚线预览框，避免用户误以为松手后没有变化。
- 四边/四角缩放光标、四边/四角拖拽缩放和标题栏拖动同时稳定。
- 便签模式缩放时，字体和 UI 控件只允许等比缩放，不允许横向或纵向拉伸变形。
- 便签窗口缩放、恢复上次位置、拖动和调整大小后，都必须重新约束窗口尺寸与品牌/图标拖动区域可达性。
- 便签窗口拖动过程中只允许同步弹窗位置和保存临时几何，不允许执行位置重置；拖动结束后再执行一次最终约束和持久化。
- 所有窗口 UI 控件风格随主题一致切换，不出现系统默认白色滚动条、未适配按钮、未适配弹窗或其他脱离主题的控件。
- 如果为了真透明和边缘缩放牺牲圆角或系统阴影，仍视为满足便签硬需求。
- 文档、实现和构建脚本都体现“精致便签以真透明和稳定边缘缩放优先”的技术决策。
